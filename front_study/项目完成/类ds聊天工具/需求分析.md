# 类DeepSeek AI聊天工具（安卓版） 需求分析文档

## 1. 项目概述

### 1.1 项目背景

用户需要一个类似DeepSeek的AI对话应用，支持与大型语言模型进行自然语言交互，核心交互体验为**流式逐字输出**，并优先交付**安卓客户端**。项目采用**UniApp框架**开发，以便未来快速拓展至iOS、小程序等多端。

### 1.2 项目目标

- 实现安卓端AI聊天工具，提供流畅的对话体验。
    
- 后端服务通过流式接口（SSE/WebSocket）传输文字，前端实时渲染打字机效果。
    
- 界面风格简洁现代化，支持多轮对话、历史记录等基础功能。
    
- 基于UniApp开发，确保在主流安卓设备上的兼容性与性能。
    

### 1.3 目标用户

- 普通用户：寻求问答、写作、翻译、编程辅助等AI对话服务。
    
- 技术爱好者：关注流式响应速度与交互细节。
    
- 移动端优先用户：习惯在手机上使用AI助手。
    

## 2. 功能性需求

### 2.1 核心功能：流式文字聊天

- **流式消息接收**：前端通过HTTP SSE（Server-Sent Events）或WebSocket与后端建立长连接，持续接收模型生成的文字片段（delta），并在对话气泡中逐字追加显示。
    
- **打字机效果**：新消息气泡在接收到第一个token时创建，后续文本平滑追加，光标闪烁效果（可选）。支持Markdown内容的逐步渲染。
    
- **中断生成**：用户可点击“停止生成”按钮，前端主动断开流式连接或发送取消请求，后端停止推理。
    
- **重试/编辑消息**：对最后一条用户消息可重新生成回答，或编辑后重新发送。
    

### 2.2 对话管理

- **新建对话**：点击新建按钮，创建空白会话，自动生成对话标题（基于首条消息摘要）。
    
- **对话列表**：侧边栏或独立页面展示历史对话列表，按时间倒序，显示标题和最后一条消息预览。支持长按删除、重命名对话。
    
- **多轮对话上下文**：将历史消息（用户+助手）作为上下文发送给API，维持连贯对话。
    
- **对话持久化**：本地存储对话记录（mysql），支持离线查看历史（无法发消息），并可同步到云端（可选）。
    

### 2.3 消息内容与交互

- **Markdown渲染**：支持标题、列表、粗斜体、表格、代码块（语法高亮）、LaTeX数学公式（如KaTeX）等。
    
- **代码块操作**：一键复制代码、显示语言标签。
    
- **长按菜单**：消息气泡支持长按复制全文、复制选中文本、分享、删除等。
    
- **消息状态显示**：发送中、流式生成中（动画）、生成完毕、发送失败、网络中断等状态清晰可见。
    

### 2.4 输入与发送

- **底部输入栏**：多行文本输入，自适应高度，最大高度限制。
    
- **发送按钮**：支持键盘回车发送（可设置），输入内容为空时禁用。
    
- **辅助功能**：语音输入（调起系统语音识别或集成第三方）、清空输入框。
    
- **附件/图片**（扩展需求）：本阶段可暂不实现，但需预留扩展接口。
    

### 2.5 用户与设置

- **API配置**：允许用户配置自定义后端API地址、API Key、模型参数（temperature、max_tokens等），并将配置安全存储在本地。

## 3. 非功能性需求

### 3.1 性能要求

- **流式延迟感知**：从发送消息到第一个token出现时间应小于2秒（在网络正常时），后续token追加间隔保持流畅。
    
- **列表性能**：对话消息较多（>100条）时，使用虚拟列表或优化渲染，避免卡顿和内存溢出。
    
- **应用启动速度**：冷启动时间不超过3秒。
    
- **包体积**：安卓APK体积控制在15MB以内（UniApp基础包+资源），后续可优化。
    

### 3.2 兼容性

- **安卓版本**：最低支持Android 7.0（API 24），覆盖主流设备。
    
- **屏幕适配**：全面屏、刘海屏、折叠屏等异形屏需正确适配安全区域。
    
- **UniApp兼容**：使用Vue 2或Vue 3（按UniApp稳定版选择），插件选择兼容安卓的API。
    

### 3.3 安全性

- **API Key安全存储**：使用平台安全存储（如Android Keystore或plus.storage加密），不得明文暴露。
    
- **网络传输安全**：强制使用HTTPS/ WSS，防止中间人攻击。
    
- **输入过滤**：前端对用户输入做基本XSS防护，但主要依赖后端安全处理。
    
- **权限最小化**：仅申请网络权限、存储权限（保存历史记录或日志），如集成语音需麦克风权限，需动态申请。
    

### 3.4 稳定性与异常处理

- **网络异常处理**：断网时提示用户，自动重连/重新请求。流式传输中断时允许重试。
    
- **错误处理**：后端返回错误（如429限流、500错误）时，前端展示明确错误提示，而不是白屏或崩溃。
    
- **崩溃日志**：集成错误收集（如Sentry或uni统计），便于问题追踪。
    

### 3.5 可维护性与扩展性

- **模块化设计**：API服务、对话管理、存储层分离，便于更换后端或UI。
    
- **多端复用**：核心业务逻辑（如SSE解析、消息处理）抽象为通用JS模块，UI层使用UniApp组件，方便未来编译至iOS、H5、微信小程序等。
    
- **国际化准备**：文案统一管理，但V1版本仅支持中文/英文。
    

## 4. 界面与交互设计（UI/UX）

### 4.1 整体布局

- **主界面**：经典聊天布局，顶部导航栏（标题、新建/侧边栏按钮），中间消息列表，底部输入栏。
    
- **侧滑抽屉/侧边栏**：展示对话历史列表、设置入口、用户头像等（可选tabBar方式）。
    
- **设置页**：独立的设置表单页面。
    

### 4.2 视觉风格

- 简洁、现代、类即时通讯工具风格。
    
- 气泡样式：用户与AI双色区分，圆角适当。
    
- 流式生成时，AI气泡带有思考动效（如闪烁的光标或三个点动画）。
    
- 支持深色模式，颜色方案参考DeepSeek（深灰蓝+亮色强调）。
    

### 4.3 动画与反馈

- 消息发送时，用户气泡从输入栏飞入列表（轻过渡）。
    
- 列表自动滚动至最新消息，若用户在浏览历史且流式输出中，提供“回到底部”悬浮按钮。
    
- 按钮点击有触觉反馈（调用Haptic API）。
    

## 5. 技术架构与数据流

### 5.1 技术选型

- **前端框架**：uni-app (Vue 3 推荐)
    
- **状态管理**：Pinia（Vue 3）或Vuex
    
- **网络请求**：uni.request 与 uni.connectSocket 或专用SSE库；注意uni-app对EventSource的支持有限，需用SocketTask或通过插件polyfill实现流式读取。
    
- **数据持久化**：uni-app的 `uni.storage` 或使用SQLite（通过插件）存储大量对话消息。
    
- **Markdown渲染**：使用 `towxml`、`marked` 或基于web-view的渲染方案，需优化渐进渲染性能。代码高亮用 `highlight.js` 或 Prism。
    
- **数学公式**：KaTeX 或 MathJax（需考虑在nvue/view中的兼容性，可能需用web-view组件）。
    

### 5.2 流式数据传输方案

**推荐方案：HTTP SSE 模拟长连接**

- UniApp端使用 `request` 加 `responseType: 'text'` 并监听 `onChunkReceived` 回调（需安卓平台支持，需测试）。或者通过插件 `uni-read-offset` 和 `plus.net.XMLHttpRequest` 实现流式接收。
    
- 后端返回标准SSE格式：`data: {"delta":"文","finish_reason":null}\n\n`。
    
- 前端解析数据块，提取delta，追加到消息content。
    

**备选方案：WebSocket**

- 更通用，双向通信，但需自行维护协议格式。
    
- 使用 `uni.connectSocket` 建立长连接，兼容性好，推荐作为最终实现方式。消息格式也可自定义为JSON，包含 `type: 'delta'` 等。
    

**协议定义（示例）**：

json

// 请求
{
  "messages": [...],
  "stream": true,
  "model": "deepseek-chat"
}
// SSE/WS 响应帧
{"id":"xxx","object":"chat.completion.chunk","choices":[{"delta":{"content":"你好"}}]}

### 5.3 数据模型

**对话（Conversation）**

- id: 本地唯一ID
    
- title: 标题
    
- createTime: 时间戳
    
- updateTime: 时间戳
    
- systemPrompt: 系统提示词（可选）
    

**消息（Message）**

- id: 本地唯一ID
    
- conversationId: 关联对话ID
    
- role: 'user' | 'assistant' | 'system'
    
- content: 文本内容（增量更新）
    
- status: 'pending'|'streaming'|'done'|'error'
    
- createTime: 时间戳
    
- tokensUsed: 消耗的token数（可选）
    

**设置（Settings）**

- apiBaseURL: 后端地址
    
- apiKey: 加密存储
    
- modelParameters: {temperature, max_tokens, top_p...}
    
- theme: 'light'|'dark'|'auto'
    

### 5.4 模块划分

- **api module**：封装WebSocket连接、流式数据解析、重连机制。
    
- **store**：管理对话列表、当前对话消息、用户设置等全局状态。
    
- **components**：消息气泡、输入栏、侧边栏、Markdown渲染器、代码块等。
    
- **pages**：主聊天页面、对话列表抽屉、设置页面。
    

## 6. 实施考虑与风险

### 6.1 UniApp流式传输兼容性风险

- Android平台需实际测试 `onChunkReceived` 兼容性，若不合规，优先采用WebSocket方案，确保可靠。
    
- WebSocket长时间连接需处理后台切换导致断开的问题，App前后台切换时自动重连。
    
- 消息大量时的渲染性能，使用虚拟列表优化长列表，避免WebView卡顿。
    

### 6.2 后端依赖

- 本分析假定后端已提供类OpenAI的流式API。需求分析仅覆盖客户端，但需定义清晰的接口规范。
    

### 6.3 迭代计划（可选）

- V1.0：单聊流式对话、Markdown基础渲染、对话历史、深色模式、API配置。
    
- V1.1：语音输入、多模态（图片理解）、对话搜索。
    
- V1.2：插件市场、自定义助手人格市场。
